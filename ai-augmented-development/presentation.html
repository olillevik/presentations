How We Work

Context is everything


The more the AI can see,
the better it performs


But context is also finite

The AI has a limited window. What's in that window
determines the quality of the output.

Most important lesson. AI is only as good as what it can see. But context is finite — working memory. Too much in and the output gets worse. Be smart about what's loaded when.

Give it access to everything

  • Code — repos, PRs, code search across repos
  • Tickets — the work to be done, requirements, context
  • CI/CD — build output, deployment status
  • Logs & monitoring — what's happening in production
  • Databases — schema, data context
  • Confluence — documentation, decisions, architecture
AI can access almost everything a developer can. Turning point was ticket access — before that, every task started with me explaining what to build. After: "we're solving ticket-1234" and it reads the requirements itself. Logs, metrics, CI, database. Each integration improves the chance of a successful outcome.

...with eyes open

Read vs. write

Can it see the database? Good.
Can it drop tables in production? Think twice.

Sandboxing

Separate contexts by capability.
The one reading prod logs shouldn't
also be fetching from the internet.


Security is a big topic — secrets, data leakage, prompt injection, access control.
Calibrate to your project's risk appetite.

"Access to everything" doesn't mean admin credentials and hope for the best. Least privilege — same principle as for humans. Security is a big topic — more than one slide. Be intentional. Calibrate to risk appetite. Internal PoC is different from a payment system. I wish we could sandbox me.

Managing context


Default context — as close to empty as possible

Then load on demand — /git-commit · /query-logs · /check-ticket · /wiki · /review


Keep default context tiny. Pull in the rest when needed.

There are many ways to layer context, but the principle is simple. Be very careful with what is always loaded. My preference is almost nothing in that box. Every always-loaded instruction consumes context before the task even starts. Then load the rest on demand: git rules for commits, log access for debugging, ticket context for implementation, wiki context for questions, review checklists for review. That keeps the model focused and leaves more room for the actual work. Shareable too — new dev joined, installed our commit skill, first commit matched convention perfectly. Zero onboarding for that.

Make it the hub


I stay in the agent.

Tickets. CI. Logs. Docs. Code. Tests.


You direct. It travels.

This is what I want in practice. I do not want to keep bouncing between tickets, logs, CI, docs, and terminals if the agent can do that for me. I stay in the agent CLI and give direction. Then it goes out, gathers context, investigates, edits, runs checks, and comes back. The point is not just automation. The point is keeping my attention in one place while the agent moves around the system.

Augmented me

  • I decide what good looks like
  • It does the legwork
  • I review and own the outcome
This is how I think about it. Not humans versus AI. Not replacement. Augmentation. I decide what we are trying to achieve and what good looks like. The AI does the legwork — reads, searches, edits, runs, checks, iterates. Then I review the result, make the judgment calls, and own the outcome. That is the division of labor.

The bottleneck shifts


As coding gets faster,
the next constraint becomes the bottleneck


Review. CI. Context. Handoffs. Approvals.

Coding is only one part of delivery. If AI makes coding faster, we do not become that much faster overall. We hit the next constraint: review, CI, approvals, missing context, handoffs. And as AI starts helping with those too, the bottleneck moves again. So the goal is not to optimize just one step. It is to iteratively improve the whole system as the bottleneck shifts.

Getting Started

Start now


Don't wait for the "right" tool

Don't wait for the "right" time

Don't wait for permission to start learning


The learning curve is real

You get measurably better at working with AI
over weeks and months. Start the clock now.

Learning curve is real. Difference between month one and month three was dramatic. Start the clock today. First step: install an AI coding tool, open your project, ask it to explain a module you didn't write. That's your Monday morning.

Working with AI

You

  • Be specific — vague prompts, vague results
  • Give context — point at the right files, explain why
  • Iterate — first attempt is rarely perfect
  • Review carefully — always verify the output

Your project

  • Think about context layering early
  • Build skills for repeated tasks
  • Set up integrations — tickets, CI, logs
  • Iterate on instructions — living documents
Two levels. You: be specific — "fix the bug" is bad, "the /api/orders endpoint returns 500 when cart is empty, look at OrderService.ts" is good. Iterate. Review carefully — output can look confident and be wrong. Project: context layering, skills for repeated tasks, integrations early. All living documents — keep improving.

What your project needs

  • Budget for AI tooling — this is not the place to save money
  • Freedom to choose tools — let the team decide what works
  • Room to experiment — some experiments won't pan out, and that's fine
  • Frictionless access — no approval forms, no usage caps

Advocate for these on your project.

Budget: cost of the tool is a fraction of the developer using it. Tool freedom: what works for one person might not work for another. Experimentation: a colleague spent $450 in two hours on an AI pipeline — called it "Gas Town." Worse results than Copilot. Team applauded the experiment. That's the culture you want. Friction kills adoption. Design governance that enables. Approve tools, set a budget, review quarterly. Don't make people fill in forms.

Backup: Hallucination & trust

AI gets things wrong — confidently. This hasn't gone away.

Mitigation — tests, CI, code review, type systems. The same safety nets that catch human bugs catch AI bugs.

The trend — hallucination rates are dropping with each model generation. But review remains essential regardless of who wrote the code.

AI hallucinates — writes code that looks correct and isn't. Mitigation: tests, CI, type checking, code review — same gates that catch human bugs. Review is review, regardless of who wrote the code.

Backup: Junior developers

The concern — "If AI writes the code, how do juniors learn?"

The shift — juniors still learn by reading, reviewing, and understanding code. AI changes who writes it, not who needs to understand it.

The opportunity — juniors working with AI get exposed to more patterns, faster. They review more code than they could write. The learning curve changes shape, it doesn't disappear.

Juniors still need to understand code to review it, debug it, and direct the AI. AI accelerates exposure — more patterns in a week than they'd write in a month. Skill shifts from "can you write it" to "can you evaluate whether it's right." Higher-order skill, not lower.
