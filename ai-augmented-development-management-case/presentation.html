Lessons from the Trenches

This is not a strategy deck and not a tooling demo. It is a real case from day-to-day software delivery. I still work close to the craft, so this is grounded in what actually happened, what changed, and what seems to matter.
Why This Matters

I do not write code by hand anymore

Since November, my work has shifted from typing code to guiding and evaluating.

real tickets real codebases real review loops real delivery constraints
The key message here is not hype. The shift is practical. This is now part of everyday engineering work: investigating incidents, implementing changes, reviewing code, and moving through delivery.
What Changed

It is no longer just autocomplete

Then

Suggest the next line or the next function.

to

Now

Read context, plan work, edit files, run checks, iterate.

For non-developers, this distinction matters. We are no longer talking about a typing assistant. We are talking about tools that can work through multi-step tasks with supervision.
How The Case Works

Three things converging

Better models

Spot a bug to navigate a repo to resolve tickets end-to-end. Each step incremental. The compound effect is not.

Better tools

Agents can navigate repos, run tests, inspect CI, and work in loops.

Better usage

We've learned how to work with these tools effectively.

None of these alone are enough. The case becomes compelling because all three are improving at the same time. Better models reward better practices more and more.
What Made It Useful

Context made the difference

The more relevant context the AI can access, the more useful it becomes.

  • Code - repositories, pull requests, code search
  • Work context - tickets, requirements, architecture notes
  • Delivery signals - CI output, logs, monitoring, schemas
This is probably the most important practical lesson. The quality of the result depends heavily on what the system can see. When the tool can read the ticket, the code, the CI output, and the logs, the interaction changes completely.
What Improved

Where it was most useful

Investigation

Faster first-pass debugging, more paths explored, less manual searching.

Implementation

Routine coding moves faster when the human directs and reviews.

Review

AI is good at consistent checklist work; humans stay focused on judgment.

I would keep this grounded. Not everything is faster, and not every task gets easier. But there is clear value in repeated engineering activities where context exists and quality gates are already in place.
Division Of Labor

What changed for the developer

Human role

Frame the problem, define what good looks like, review the output, own the result.

+

AI role

Do the legwork: read, search, draft, edit, run checks, and iterate.

This case points more toward augmentation than toward fully autonomous delivery.

This is an important framing point for management audiences. The most useful model today is not replacement. It is leverage. Human accountability remains intact, but the distribution of effort changes.
What Did Not Work

Full autonomy was less useful than expected

Easy tasks were often faster directly from my own AI-assisted workflow. Hard tasks still needed close human involvement. That left less room in the middle than expected.

So the stronger case today is not "let the system run alone". It is "give skilled people much more leverage".

This is a useful credibility slide because it shows restraint. We did test the more ambitious idea. The lesson was not that autonomy never matters, but that today the practical value is more visible in augmentation.
Operational Reality

The bottleneck moves

When coding speeds up, the next constraint becomes more visible.

review CI handoffs missing context approvals
This is where the case becomes broader than coding. If one part of the system speeds up, another part becomes the limiting factor. That is why the impact is real, but uneven.
Conditions Around The Work

What developers needed for this to work

  • Access - tools and relevant systems had to be reachable
  • Low friction - learning stalls when usage is buried in process
  • Room to experiment - skill grows through repeated practical use
  • Guardrails - useful access is not the same as unrestricted access
This keeps the focus on the case rather than turning into a decision memo. These were simply the conditions that made the case possible and useful.
Main Takeaways

Takeaways

Context creates leverage

Access, integrations, and context layering make AI useful.

Humans with AI beat either alone

Today, augmentation is more valuable than autonomy.

Improve the whole process

As coding gets faster, review, CI, context, and handoffs matter more.

If the audience remembers only three things, these are probably the right three. This was not a gimmick, not a silver bullet, and not a replacement story. It was a practical shift in how software work gets done.
Closing

My conclusion from the case

The important question is no longer whether AI belongs in software development. It already does.

The real variation is in how effectively different teams learn to work with it.

This gives you a clean close without turning the talk into a management mandate. It stays descriptive, grounded, and aligned with your voice from the engineering floor.
